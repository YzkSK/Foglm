# 📷 仕様書（アプリ名：Foglm）

一緒に撮った写真が「忘れた頃」に全員へ同時に届く、フィルムカメラ風の共有アルバムアプリ

> ※ アプリ名は **Foglm** に確定（リポジトリ：[YzkSK/Foglm](https://github.com/YzkSK/Foglm)）

---

## 1. 概要

### 1.1 コンセプト
グループ内で1つのフィルムを共有して写真を撮り、その日の写真の中からグループ投票で選ばれた1枚だけがその日のうちに現像される。選ばれなかった写真は、3〜14日後のランダムなタイミングで「忘れた頃」に全員へ同時に届く。

### 1.2 ターゲット
SNSでの「盛る」ことに疲れた中高生・大学生。少人数グループ（2〜6人）での親密な共有を前提とする。

### 1.3 コア体験（1日の流れ）
1. **日中 — 撮る**：グループで共有フィルムを使って撮影する（1日上限10枚、グループ全体で共有）
2. **随時 — 投票する**：撮影された写真はボヤけた状態で一覧に並び、メンバーが「今日の1枚」に投票する（何度でも投票し直せる）
3. **24時 — 締め切り／現像**：投票が締め切られ、最多得票の1枚がその場で現像される。他の写真は今後3〜14日のランダムな日に現像される
4. **後日 — 忘れた頃に届く**：ランダム待ちだった写真が、対象日になった時点で全員に同時に鮮明化されて届く

---

## 2. 用語定義

| 用語 | 意味 |
|---|---|
| フィルム | グループが1日に撮影できる写真の上限枚数（共有プール） |
| 現像 | ボヤけた写真が鮮明な状態になり、閲覧可能になること |
| 今日の1枚 | その日撮影された写真の中から投票で選ばれ、即日現像される写真 |
| ランダム現像 | 今日の1枚に選ばれなかった写真が、3〜14日後のランダムな日に現像されること |
| 現像待ち | まだ現像されていない、ボヤけた状態の写真 |

## 3. 機能仕様

### 3.1 認証・アカウント
- ログインは **SNSログイン（Google / Apple / X / Instagram など）**に加え、**メール・パスワード方式**も採用する
- **メール・パスワード方式の詳細仕様**
  - メールアドレスの確認は **必須**。確認メール内のリンクを踏むまでは本人確認未完了の状態とし、ログイン後の主要機能（撮影・投票など）を利用不可にする（**この確認フローはメール・パスワード方式でのアカウントにのみ適用され、SNSログインのアカウントは対象外**）
  - パスワード要件：**8文字以上、かつ英大文字・英小文字・数字を全て含むこと**を必須とする（クライアント側での入力チェックに加え、サーバー側でも検証する）
  - パスワードリセットは **メール内のリセットリンクを踏む方式**とする（確認コード入力方式は採用しない）
  - **アカウント重複防止**：SNSログイン（Google / Apple / X / Instagram）で使用されているメールアドレスと同一のアドレスでメール・パスワード方式の新規登録が行われた場合、登録を **ブロック**し、「そのメールアドレスは〇〇（既存の認証方法）で登録済みです」と案内して既存のログイン方法へ誘導する。逆に、メール・パスワード方式で登録済みのアドレスと同一のメールアドレスでSNSログインが行われた場合も同様にブロックし、メール・パスワード方式でのログインを案内する
- 初回ログイン時にニックネーム・アイコンを設定する

#### 3.1.1 ログアウト
- 設定・マイページ画面（S12）から、いつでもログアウトできる
- ログアウトすると認証セッションを破棄し、ログイン画面（S01）に戻る

#### 3.1.2 プロフィール編集
- 設定・マイページ画面（S12）から、**ニックネーム・アイコンをいつでも変更**できる
- 変更内容は、変更後に投稿するリアクション・コメント等から即座に反映される（過去の投稿の表示名も、表示時に最新のプロフィールを参照するため自動的に更新される）

#### 3.1.3 アカウント削除
- 設定・マイページ画面（S12）から、アカウント削除を実行できる
- 削除方式は **ソフト削除**とする
  - `users` 行は物理削除せず、`deleted_at` を設定した上で個人情報（表示名・アイコン・メールアドレス・SNS連携情報）を**匿名化**する（例：表示名を「退会したユーザー」に置き換え）
  - グループ内の写真・撮影者情報・投票履歴・リアクション・コメントなど、**他メンバーに関わるデータは壊さない**（参照先の `users` 行は匿名化された状態で残るため、整合性は保たれる）
- **所属していた固定グループ・イベントグループ**については、それぞれについて **「脱退」と同一の処理を一括実行**する（3.2.1章の `leave_group` ロジックをグループごとに適用。作成者権限の自動委譲、残り1人になった場合の猶予期間、メンバー0人時の解散なども通常の脱退と同様に連動する）
- **ソロモードのグループ・写真**は、本人以外が利用しない個人データのため、アカウント削除と同時に **完全に削除**する（ソフト削除の対象外）
- アカウント削除は取り消せない旨を、実行前に確認画面で警告する
- **削除後の再ログイン**：削除済みアカウント（`deleted_at` が設定されたアカウント）は、**同じログイン手段での再ログインを完全にブロック**する
  - ログイン試行時に `users.deleted_at IS NOT NULL` を検知した場合、認証自体は成立してもアプリ側で即座にサインアウトさせ、「このアカウントは削除済みのため、ご利用いただけません」と案内する
  - 同じSNSアカウント・同じメールアドレスでの再利用は不可。改めて利用したい場合は、**別のログイン手段（別のメールアドレス・別のSNSアカウントなど）で新規登録**する必要がある
  - 補足：`users` テーブル上の `email` 列はアプリ側の表示用データとして匿名化されるが、**Supabase Auth側の認証情報（そのメールアドレス・SNSアカウントに紐づくログインID自体）は削除しない**。そのため、匿名化後も同じメールアドレスでの新規サインアップやSNS再ログインは、Supabase Auth側の重複チェックによって引き続きブロックされる

### 3.2 グループ
- 1グループの人数は **2人〜6人**とする（1人での利用は「3.10 ソロモード」で独立した仕組みとして提供する）
- グループの新規作成・招待は **メンバー全員が可能**（作成者限定にしない）
- 招待は招待リンク／招待コードの共有によって行う

#### 3.2.1 脱退・解散ルール
**脱退**は固定グループ・イベントグループの **両方**に対応する（画面S13は共通）。**解散**および「残り1人になった場合の猶予期間」は **固定グループのみ**を対象とする（イベントグループは3.11章の「終了日到来による自動クローズ」を持つため、解散機能や猶予期間は設けない）。

- **脱退**：メンバーは各自いつでもグループを脱退できる（画面はS13）
  - 脱退した時点で、そのグループに関する写真の **閲覧権限を失う**
  - 脱退以降、そのグループの写真は一切閲覧できなくなる（脱退前に見えていた写真も、脱退後は非表示になる）
  - **残り1人になった場合（固定グループのみ）**：現役メンバーが1人になった時点から **1週間の猶予期間**を設ける。猶予期間中に新たなメンバーが加入して2人以上に戻れば通常運用を継続する。1週間経っても1人のままだった場合、そのグループは自動的に **アーカイブ状態**へ移行する
    - アーカイブ状態は、イベントグループのクローズと同様の扱いとする（新規の撮影・投票・新規メンバー招待は不可。閲覧・リアクション・コメントの追加は引き続き可能）
    - 残り1人の状態から、さらにその1人も脱退した場合（メンバー0人）は、猶予期間を待たず **即座に解散**する（写真・データは全て完全削除される）
    - **新規作成直後で、まだ一度も2人以上になったことがない固定グループ（メンバー1人のまま）は、この猶予期間ルールの対象外**とする（`solo_since` は脱退によってメンバーが1人に減った場合にのみ設定され、グループ作成時には設定しない）
  - **作成者が脱退した場合（固定グループのみ）**：脱退と同時に、残っている現役メンバーの中から **ランダムに選ばれた1人**へ、作成者としての権限（解散権限を含む）を自動的に委譲する
    - メンバーが自分のみ（脱退により0人になる場合）は、権限委譲は発生せず「メンバー0人」のルール（即座に解散）が優先される
    - イベントグループには解散機能自体が存在しないため、**作成者脱退時の権限委譲は行わない**（`created_by` はそのまま記録として残す）
- **解散（固定グループのみ）**：**グループ作成者のみ**が解散を実行できる
  - 解散すると、そのグループに属する **写真・投票データ・リアクション・コメントなど全てのデータを完全に削除**する（復元不可）
  - 解散後、全メンバーは即座にグループへアクセスできなくなる

### 3.3 フィルム（撮影枚数）仕様
- フィルムは **グループ全体で1プールを共有**する（誰が撮影しても同じ枚数が減る。メンバー個別の割り当てではない）
- 1日の上限は **10枚**（グループ全体）
- 上限は **毎日リセット**される（日次）
- その日に使い切らなかった枚数は **繰り越さず消える**
- 課金要素は **現時点では設けない**（将来的な検討事項とする）

### 3.4 撮影
- 撮影方法は以下の両方に対応する
  - 各メンバーが自分の端末でそれぞれ撮る
  - 代表者1人が撮影し、その写真が全員に共有される
- 撮影された写真は、撮影直後に **鮮明な原本をサーバーの非公開領域へ送信**し、端末・一覧には **ボヤけた版のみ**が表示される

### 3.5 「今日の1枚」投票
- 対象：その日（0:00〜24:00）にグループ内で撮影された全ての写真
- 表示：投票中も写真は **ボヤけたまま**（うっすらとしか見えない状態）で提示される
- 投票方法：グループの多数決。**1人につき投票権は1票**とし、その1票を**何度でも投票し直せる（最後の一票のみが有効）**
- 締め切り：**毎日24時（日本時間、日付が変わる直前）**
- 集計：締め切り時点で最多得票の写真を「今日の1枚」とする
- 同数の場合：**同数だった写真の中からランダムで1枚**を選出する
- **投票が1票も入らなかった場合**：その日撮影された写真の中から **ランダムで1枚**を選出し、即日現像する（今日の1枚が0件になることを防ぐ）
- その日、写真が1枚も撮影されなかった場合は投票自体が発生しない

### 3.6 現像フロー
| 種別 | 対象 | 現像タイミング |
|---|---|---|
| 即日現像 | 今日の1枚に選ばれた写真 | 投票締め切り（24時）と同時 |
| ランダム現像 | 選ばれなかった残りの写真 | 撮影日から **3〜14日後のランダムな日** |

- どちらの現像も、**一緒に撮った（同じグループの）全員に同時に**公開される
- 現像前はボヤけた状態、現像日に **一気に鮮明な状態へ切り替わる**

### 3.7 現像後の体験
- 現像された写真には **リアクション（絵文字など）とコメント**を付けられる
- 写真は **撮影日時が分かる形**でアルバムに残り続ける
- 気に入った写真は **スマホ端末に保存**できる

### 3.8 通知・継続利用の仕掛け
- **プッシュ通知**は以下の1種類のみに絞る（過剰通知を避ける）
  1. 現像完了時の通知（今日の1枚／ランダム現像 共通）
- **画面内表示**（プッシュ通知は使わない）は以下の2つを行う
  1. 現在の現像待ち枚数の表示
  2. 投票を促す表示（当日の候補写真がある場合に、グループホーム等の画面内でのみ案内する）
- 上記以外のプッシュ通知（投票を促すプッシュ通知など）は設けない

### 3.9 世界観・デザイン方針
- フィルムカメラ／使い捨てカメラを想起させる **フィルム風・レトロ**なビジュアル

### 3.10 ソロモード
グループとは別に、**1人での日記的な利用**に対応する独立したモード。

- **発行方法**：全ユーザーに **自動で付与**される。サインアップ（初回ログイン）と同時に、本人専用の「ソロモード空間」が作成され、招待や作成操作は不要
- **利用の流れ**：グループモードと **同じ画面・同じ流れ**を再利用する
  - 撮影 → 候補一覧（ボヤけ表示）→ 「今日の1枚」を自分で選ぶ → **毎日24時に現像**（グループモードと同じ締め切り・待つ体験を維持）
  - 選ばなかった残りの写真は、グループモードと同様に **3〜14日後のランダムな日**に現像される
  - 投票者が本人1人のみのため、選んだ写真がそのまま「今日の1枚」として確定する（グループモードの集計ロジックをそのまま流用でき、同数判定は発生しない）
- **撮影上限**：ソロモードは個人利用のため、グループモードの10枚より少ない **1日3枚**に設定する
- **リアクション・コメント**：ソロモードでは提供しない（自分のみが閲覧するため不要と判断）
- **友人・グループへのシェア機能**：**提供しない**（ソロモードは個人の日記的な利用に閉じた機能とする）
- ソロモードのアルバムとグループのアルバムは、UI上ではグループ一覧画面（S03）に **「自分」枠として常時1つ表示**する形で区別する

### 3.11 イベントグループ
固定グループ（3.2）とは別に、**特定のイベント（旅行・飲み会など）単位で一時的に使う**グループ種別。「たまに混ざる人」を都度招待しやすくし、固定グループの人間関係に依存しない使い方を提供する。

- **作成者**：**アプリ利用者なら誰でも**自由に作成・参加できる（固定グループのメンバーに限定しない）
- **期間設定**：作成時に **開始日・終了日（期間）を指定**する（例：1日限定の集まり、3日間の旅行など）
- **撮影・投票ルール**：固定グループと **同一**（1日10枚をグループ全体で共有、ボヤけたまま多数決で「今日の1枚」を選出、締め切りは毎日24時、0票の場合はランダム選出）
- **人数上限**：固定グループと同じ **2〜6人**
- **終了後の扱い**：指定した終了日を迎えると自動的に **アルバム化してクローズ**する
  - クローズ後は新規の撮影・投票・新規メンバー招待はできない
  - ただし、終了時点でまだランダム現像待ちだった写真は、**予定通り現像日にそのまま現像される**（現像スケジュール自体はクローズによって止めない）
  - クローズ後も **リアクション・コメントの追加投稿は可能**（後から思い出を追加して楽しめる）

```
                                   ├→ S09 アルバム → S10 写真詳細（クローズ後も閲覧・リアクション・コメント可）
                                   ├→ S11 現像待ちステータス（常時表示・バッジ）
                                   └→ S13 グループ脱退確認 → 脱退実行後 S03へ戻る
```

### 5.2 フィルム上限（撮影枚数）の制御

専用テーブルは持たず、`photos` テーブルにおける `group_id` × `taken_date` の件数をカウントして上限を判定する。上限枚数は `groups.mode` によって出し分ける。

| モード | 1日の上限 |
|---|---|
| group（固定グループ） | 10枚（グループ全体で共有） |
| event（イベントグループ） | 10枚（グループ全体で共有、固定グループと同一ルール） |
| solo（本人専用） | 3枚 |

#### 5.2.1 `taken_date` の算出基準
- `taken_date` は **日本時間（Asia/Tokyo）を基準に算出した日付**とする（`taken_at`（UTC保存のtimestamptz）を `AT TIME ZONE 'Asia/Tokyo'` で変換した日付部分）
- これは「今日の1枚」投票の締め切り（毎日24時＝日本時間の日付が変わるタイミング）と同じ基準に揃えることで、**フィルム消費の1日単位と投票対象の1日単位を完全に一致させる**ため

#### 5.2.2 排他制御（同時撮影による上限超過の防止）
複数メンバーがほぼ同時に撮影した場合、単純な「件数チェック→INSERT」では **競合（上限を超えて登録されてしまう）が発生しうる**。これを防ぐため、以下の方式でDB側で排他制御を行う。

- `photos` へのINSERTは、**PostgreSQLのトランザクション内で `pg_advisory_xact_lock`（`group_id` と `taken_date` から生成したキーによるアドバイザリロック）を取得した上で実行**する
- ロック取得後に該当 `group_id` × `taken_date` の件数を再カウントし、上限未満であることを確認してから INSERT する
- トランザクション終了時にロックは自動解放される
- これにより、同時に撮影リクエストが来ても **1件ずつ順番に処理**され、上限を超える登録は発生しない

#### 5.2.3 上限到達時の挙動
- 上限に達している状態でのアップロードリクエストは、Edge Function（`upload_photo`）が **エラーを返し、写真は保存しない**（原本・ボヤけ版とも生成しない）
- カメラ撮影画面（S06）は、`get_today_shots_remaining` の残り枚数を **リアルタイムに表示・購読**し、残り0枚になった時点で **シャッターボタンを操作不可（グレーアウト）**にする
- 残り枚数の表示・シャッター制御には Supabase Realtime を用いて、他メンバーの撮影によって残数が減った場合も即座に画面へ反映する

| `process_scheduled_development`（cron） | Edge Function | 1時間ごとに実行。`develop_scheduled_at` が到来した写真を `developed` に更新し、原本へのアクセスを解放。**通知は写真単位ではなく、同一グループ・同一実行タイミングでまとめて集計し、1件のプッシュ通知（例：「3枚の写真が現像されました」）として対象グループ全員へ送信する**（1枚のみの場合はその旨の文言にする） |
| `get_album` | Query | 現像済み写真をグループ・撮影日順に取得 |
| `get_developing_count` | Query | 現在の現像待ち枚数を取得（ホーム画面のバッジ表示用） |

### 6.6 リアクション・コメント
| API | 種別 | 概要 |
|---|---|---|
| `add_reaction` | RPC | 現像済み写真へのリアクション追加 |
| `add_comment` | RPC | 現像済み写真へのコメント追加 |

### 6.7 スケジューラ構成
- `close_daily_vote`：**Supabase Cron（Scheduled Functions）** で毎日 **日本時間24:00（＝UTC 15:00）**に起動
- `process_scheduled_development`：**Supabase Cron（Scheduled Functions）** で1時間おきに起動し、対象の写真を随時現像
- `archive_expired_events`：**Supabase Cron（Scheduled Functions）** で毎日起動し、終了日を迎えたイベントグループをアルバム化・クローズ
- `archive_inactive_solo_groups`：**Supabase Cron（Scheduled Functions）** で毎日起動し、残り1人の状態が1週間続いた固定グループをアーカイブ

## 7. 通知仕様

### 7.1 プッシュ通知（FCM）
| トリガー | 通知内容 | 送信先 |
|---|---|---|
| 今日の1枚が現像された | 「今日の1枚が現像されました」＋サムネイル | 該当グループの全メンバー |
| ランダム現像が完了した | 「〇日前の写真が現像されました」＋サムネイル（**同一実行タイミングで複数枚が現像された場合は「N枚の写真が現像されました」とまとめて1件に集約**） | 該当グループの全メンバー |

- 送信方式：**FCM（Firebase Cloud Messaging）**。Supabase Edge Function から FCM API を呼び出す
- プッシュ通知はこの2種類のみに限定する（通知疲れを防ぐ設計）

### 7.2 画面内表示（プッシュ通知は使わない）
| 表示内容 | 表示条件 |
|---|---|
| 現像待ち枚数 | 常時、グループホーム等に件数を表示 |
| 投票を促す表示 | 当日の候補写真が存在し、かつ本人が未投票の場合に、グループホーム等の画面内でのみ案内する |

- 投票の呼びかけは **端末へのプッシュ通知としては送らず**、アプリを開いたときの画面内表示にとどめる

## 8. 非機能要件

### 8.1 セキュリティ（最重要）
- 撮影直後の **鮮明な原本は非公開ストレージにのみ保存**し、クライアント（アプリ）には一切配信しない
- ボヤけ画像の生成は **必ずサーバー側（Edge Function + sharp）**で行う。クライアント側での加工は行わない（原本がそのまま端末に渡ることを防ぐため）
- 現像日が来るまでは、署名付きURLもボヤけ版のみ発行する
- Supabase の **RLS（Row Level Security）** により、写真・コメント等は同一グループの **現役メンバー**（`group_members.left_at IS NULL`）のみ閲覧・操作可能とする。脱退済みメンバー（`left_at` が設定されたユーザー）は、脱退前に見えていた写真も含めて即座に閲覧不可となる

### 8.2 パフォーマンス
- 配信画像は **CDN**（Content Delivery Network。画像などのファイルを世界各地の中継サーバーにあらかじめキャッシュしておき、利用者に近い場所から高速に届ける仕組み）経由で配信し、現像タイミング（複数ユーザーへの同時公開）でのアクセス集中に備える

### 8.3 可用性
- cron処理（投票締め切り・ランダム現像判定）はサーバー側で完結させ、クライアントの起動状態やタイムゾーンに依存しない設計とする

### 8.4 データ保存期間
- 写真データ（原本・ボヤけ版とも）は **一旦無期限保存**とする。自動削除・自動アーカイブの仕組みは設けない
- 将来的にユーザー数・データ量が増えた段階で、保存期間の制限やストレージコストの見直しを改めて検討する（現時点では要件化しない）
