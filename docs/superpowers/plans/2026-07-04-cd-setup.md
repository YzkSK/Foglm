# CD構築 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** mainブランチのsupabase配下への変更を検知して、マイグレーション適用とEdge Functionsデプロイを自動化するGitHub Actionsワークフローを構築する。

**Architecture:** `.github/workflows/cd.yml` に単一ジョブ `deploy` を定義する。`supabase/setup-cli@v1` でCLIをセットアップし、`supabase link` → `supabase db push` → (Edge Functionsが存在する場合のみ)`supabase functions deploy` の順で実行する。ロールバック手順は自動化せず `docs/setup/supabase.md` に文書化する。

**Tech Stack:** GitHub Actions, `supabase/setup-cli@v1`, Supabase CLI

## Global Constraints

- 新規依存パッケージの追加はしない。
- Secretsの実際の値はチャットに含めず、登録はユーザーが行う。
- 依頼されていないファイルは変更しない。

### Task 1: CDワークフローの作成

**Files:**
- Create: `.github/workflows/cd.yml`

**Interfaces:**
- Consumes: GitHub Secrets `SUPABASE_ACCESS_TOKEN`, `SUPABASE_PROJECT_ID`, `SUPABASE_DB_PASSWORD`(ユーザーが登録)
- Produces: なし

- [ ] **Step 1: cd.ymlを作成**

`.github/workflows/cd.yml` を新規作成:
```yaml
name: CD

on:
  push:
    branches: [main]
    paths:
      - 'supabase/**'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: supabase/setup-cli@v1
        with:
          version: latest

      - name: Link Supabase project
        run: supabase link --project-ref "$SUPABASE_PROJECT_ID"
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
          SUPABASE_PROJECT_ID: ${{ secrets.SUPABASE_PROJECT_ID }}
          SUPABASE_DB_PASSWORD: ${{ secrets.SUPABASE_DB_PASSWORD }}

      - name: Push database migrations
        run: supabase db push
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
          SUPABASE_DB_PASSWORD: ${{ secrets.SUPABASE_DB_PASSWORD }}

      - name: Check for edge functions
        id: check_functions
        run: |
          if find supabase/functions -type f -name '*.ts' | grep -q .; then
            echo "has_functions=true" >> "$GITHUB_OUTPUT"
          else
            echo "has_functions=false" >> "$GITHUB_OUTPUT"
          fi

      - name: Deploy edge functions
        if: steps.check_functions.outputs.has_functions == 'true'
        run: supabase functions deploy
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
          SUPABASE_PROJECT_ID: ${{ secrets.SUPABASE_PROJECT_ID }}
```

- [ ] **Step 1: docs/setup/supabase.mdの末尾に手順を追記**

`docs/setup/supabase.md` の末尾に以下のセクションを追記する:
```markdown

## 6. デプロイ失敗時の対応(ロールバック方針)

CD(`.github/workflows/cd.yml`)は `supabase db push` / `supabase functions deploy` の自動ロールバックを行わない。失敗時は以下の手順で手動対応する。

### マイグレーション適用(`supabase db push`)が失敗した場合
1. GitHub Actionsのログでどのマイグレーションファイルが失敗したか確認する。
2. 本番環境のスキーマ状態を確認し、失敗したマイグレーションが部分適用されていないか確認する。
3. 問題のあるマイグレーションを直接編集せず、それを打ち消す新しいマイグレーションSQL(`supabase/migrations/`に追加)を作成する。
4. 修正後、再度mainにマージしてCDを再実行する。

### Edge Functionsデプロイ(`supabase functions deploy`)が失敗した場合
1. デプロイに失敗しても、直前にデプロイ済みのバージョンは稼働し続けるため、サービス断は発生しない。
2. GitHub Actionsのログでエラー内容を確認し、Edge Function側のコードを修正する。
3. 修正後、再度mainにマージしてCDを再実行する。
```

ユーザーに以下を伝える(実際の値の入力はユーザーが行う):
```bash
gh secret set SUPABASE_ACCESS_TOKEN --repo YzkSK/Foglm
gh secret set SUPABASE_PROJECT_ID --repo YzkSK/Foglm
gh secret set SUPABASE_DB_PASSWORD --repo YzkSK/Foglm
```
それぞれの値の取得場所:
- `SUPABASE_ACCESS_TOKEN`: Supabaseダッシュボード → Account → Access Tokens
- `SUPABASE_PROJECT_ID`: Supabaseダッシュボード → 対象プロジェクト → Project Settings → General → Reference ID
- `SUPABASE_DB_PASSWORD`: プロジェクト作成時に設定したデータベースパスワード

CDワークフローはこのPRのCI(`flutter-ci`)には影響しないため、通常通りマージ判断を行ってよい。
